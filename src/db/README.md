# `src/db` — Data layer (Drizzle ORM + Neon Postgres)

Implemented in **Phase 3**.

## Layout

- `client.ts` — Drizzle client on Neon's serverless HTTP driver. No
  `server-only` guard, so CLI scripts can use it.
- `index.ts` — `server-only` re-export of `client.ts`. **App code imports the
  client from `@/db`**; only scripts import `@/db/client`.
- `schema/` — table definitions per domain + `enums.ts`, `relations.ts`, shared
  column helpers (`_shared.ts`), and a barrel `index.ts`. Columns are written
  camelCase and mapped to snake_case via `casing: "snake_case"`.
- `seed.ts` / `seed-constants.ts` — idempotent seed loading the prototype
  dataset (`@/lib/data`, the typed port of `black stars html/data.js`).
- `verify.ts` — asserts the seeded totals reproduce the prototype's figures via
  the real repositories in `src/server/services`.

Migrations are generated by `drizzle-kit` into `/drizzle` at the repo root
(`drizzle.config.ts`).

## Conventions

- **Money** is whole KES shillings stored as `bigint` — see
  [ADR 0002](../../docs/decisions/0002-money-storage.md).
- **Multi-tenant:** every domain table carries `clubId` (Phase 4 scopes by
  session). **Audit:** `createdAt` / `updatedAt` (+ nullable `createdBy`) on
  every table.
- **Business night:** sales/expenses/orders carry a `businessDate` (a trading
  night runs ~8pm→4am, so it differs from the calendar date after midnight).
- The prototype's pre-aggregated arrays (`byHour`, `paymentMix`, `weekly`, …)
  are persisted as **reporting snapshot** tables; the recent sale/order feeds
  seed the transactional tables.

## Scripts

```bash
pnpm db:generate   # generate a migration from the schema
pnpm db:migrate    # apply migrations to DATABASE_URL
pnpm db:seed       # truncate + load the dev dataset (idempotent)
pnpm db:verify     # assert seeded totals match the prototype
pnpm db:studio     # open Drizzle Studio
```

Requires `DATABASE_URL` in `.env` (see `.env.example`). Scripts load it via
`--env-file=.env`.
